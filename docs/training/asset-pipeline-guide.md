# 游戏资源管线指南

## 为什么需要资源管线

代码 demo 只证明玩法能跑，不能证明游戏能做成品。

一个真正能从 0 到 1 做出来的小游戏，还需要处理：

- 图像资源从哪里来；
- 3D 模型从哪里来；
- 音效和音乐从哪里来；
- 字体能不能商用；
- 授权是否安全；
- 资源如何导入 Cocos；
- 资源如何命名、压缩、归档；
- 构建后体积是否可控；
- 后续能不能替换美术而不重写逻辑。

所以后续每个实战项目都必须同时训练两条线：

```text
玩法代码线
资源生产线
```

## 资源获取原则

优先级建议：

1. 训练阶段优先使用可商用免费资源，避免授权干扰学习。
2. 做作品集或发布版本时，再考虑购买付费资源包或定制资源。
3. 所有资源都要记录来源和授权。
4. 不清楚授权的资源不进项目。
5. 不要随手从搜索引擎图片结果里拖图进项目。

## 推荐资源来源

### 2D 图像

适合训练和原型：

- Kenney：大量 CC0 游戏素材，适合 2D、UI、音效、简单 3D。
- OpenGameArt：资源很多，但授权类型混杂，必须逐个看清授权。
- itch.io Asset Packs：适合找特定风格资源，免费和付费都有，必须看每个包的 license。

建议用法：

- 前期使用 Kenney 这类统一风格素材，快速完成可玩版本。
- 当玩法确定后，再换成更统一的美术包。

### 3D 模型

适合训练和原型：

- Quaternius：大量 CC0 低多边形模型，适合入门 3D。
- Poly Haven：CC0 HDRI、贴图、部分模型，适合环境和材质。
- Kenney：也有简单 3D 模型包。

建议用法：

- Cocos 3D 入门阶段优先使用低多边形模型。
- 单个模型先控制在较低面数，避免一开始被性能和导入问题拖住。
- 优先使用 `.glb` 或 `.gltf`，再根据 Cocos 支持情况处理材质和贴图。

### 音效

适合训练和原型：

- Kenney 音效包：授权简单，风格统一。
- Freesound：资源多，但授权类型不同，必须检查是否需要署名、是否允许商用。
- OpenGameArt：也有音效，但同样要逐个看授权。

建议用法：

- UI 点击、收集、失败、开始、胜利都应该有基础音效。
- 原型阶段先用少量短音效，不急着做复杂混音。

### 音乐

适合训练和原型：

- OpenGameArt 音乐区。
- itch.io 音乐包。
- Freesound 上的循环音乐片段。

建议用法：

- 背景音乐先用短循环。
- 音乐音量要低于音效，避免干扰反馈。
- 如果要发布，必须确认授权允许商用和再分发。

### 字体

适合训练和原型：

- Google Fonts：大量开源字体，通常适合 Web 和游戏 UI。

建议用法：

- 中文项目要特别注意字体文件体积。
- 训练项目可以先使用系统默认字体。
- 发布版本再选择明确授权的字体。

## 授权检查清单

每次下载资源都要问：

- 是否允许商用？
- 是否需要署名？
- 是否允许修改？
- 是否允许随游戏一起分发？
- 是否限制平台？
- 是否要求相同协议共享？

常见授权理解：

- CC0：最省心，通常可以商用、修改、无需署名。
- CC-BY：通常可商用，但需要署名。
- CC-BY-SA：通常需要按相同协议分享衍生内容，发布前要谨慎。
- GPL / LGPL：不建议随意用于游戏资源，除非你清楚影响。
- 自定义 license：必须读原文。

项目内建议新建资源记录：

```text
docs/assets/asset-credits.md
```

记录格式：

```md
| 资源 | 来源 | 作者 | 授权 | 用途 | 链接 |
| --- | --- | --- | --- | --- | --- |
| collect.wav | Kenney | Kenney | CC0 | 收集音效 | https://kenney.nl |
```

## Cocos 项目资源目录建议

建议后续统一使用：

```text
assets/
  scenes/
  scripts/
  prefabs/
  textures/
  audio/
    sfx/
    music/
  fonts/
  models/
  materials/
  ui/
```

规则：

- 场景放 `assets/scenes`。
- 脚本放 `assets/scripts`。
- 可复用对象放 `assets/prefabs`。
- 图片放 `assets/textures` 或 `assets/ui`。
- 音效放 `assets/audio/sfx`。
- 音乐放 `assets/audio/music`。
- 字体放 `assets/fonts`。
- 3D 模型放 `assets/models`。

## 2D 资源导入流程

基本流程：

1. 下载资源包。
2. 检查授权。
3. 解压到项目外的临时目录。
4. 挑选真正需要的图片。
5. 改名成项目风格。
6. 放进 `assets/textures` 或 `assets/ui`。
7. 回到 Cocos 等待生成 `.meta`。
8. 在 Sprite / UI 组件中引用。
9. 提交图片和 `.meta`。

命名建议：

```text
player_idle.png
star_collect.png
button_start.png
enemy_slime.png
```

避免：

```text
image (1).png
new_new_star_final.png
素材副本.png
```

## 音频导入流程

基本流程：

1. 下载音效或音乐。
2. 检查授权。
3. 优先选择短小、清晰的 `.wav` 或 `.mp3`。
4. 放入 `assets/audio/sfx` 或 `assets/audio/music`。
5. 在 Cocos 中用 AudioSource 或脚本播放。
6. 控制音量，避免刺耳。

命名建议：

```text
sfx_collect.wav
sfx_game_over.wav
sfx_click.wav
music_loop_main.mp3
```

训练阶段音频目标：

- 收集星星有音效。
- 点击按钮有音效。
- Game Over 有音效。
- 背景音乐可循环。

## 3D 模型导入流程

基本流程：

1. 优先下载 `.glb` 或 `.gltf`。
2. 检查模型授权。
3. 检查贴图是否齐全。
4. 放入 `assets/models`。
5. 回到 Cocos 等待导入。
6. 检查模型尺寸、朝向、材质。
7. 必要时用 Blender 调整尺寸、原点和导出设置。

训练阶段建议：

- 先做低多边形模型。
- 不要一开始追求写实材质。
- 单个场景少量模型，先理解相机、灯光和坐标。

## AI 生成资源的使用原则

可以用 AI 辅助生成：

- 概念图。
- 临时图标。
- UI 背景。
- 简单贴图。
- 音效草稿。

但要注意：

- AI 生成资源也要确认平台授权。
- 生成结果需要统一风格。
- 不要直接混用大量不同风格图片。
- 发布前要做版权和商用风险检查。

## 每个实战项目必须补齐的资源清单

每个实战游戏开始前，都要写：

```text
需要的 2D 图片
需要的 3D 模型
需要的音效
需要的音乐
需要的字体
资源来源
授权记录
导入目录
替换计划
```

## Star Catcher 后续资源计划

当前 Star Catcher 还主要是程序绘制图形。后续打磨时建议补：

- 星星收集音效。
- 点击 Start / Restart 音效。
- Game Over 音效。
- 简单背景音乐。
- 背景图片或简单星空纹理。
- 玩家和星星替换为图片资源或 Prefab。

建议先用免费可商用资源完成第一个版本，不急着追求原创美术。
