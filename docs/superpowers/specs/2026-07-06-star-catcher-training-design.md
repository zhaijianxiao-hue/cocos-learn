# Star Catcher Cocos 训练设计

## 目标

这个项目是一个 Cocos Creator 3.8 的教练式学习项目，面向已有多年开发经验、希望通过实战建立游戏开发直觉的学习者。

我们的目标不是背诵 Cocos 概念，也不是一开始就做一个复杂游戏。我们的目标是：通过一个小而完整的 2D 游戏，逐步理解 Cocos 的运行模型、编辑器工作流、组件组织方式和游戏循环。最终你需要能看懂每一步在做什么，并且能更高质量地和 AI 共创 Cocos 项目。

## 游戏范围

第一阶段训练游戏叫 `Star Catcher`，使用 Cocos Creator 3.8 + TypeScript 开发。

第一版完成后应该包含：

- 一个可以在可视区域内移动的玩家对象；
- 可以出现在场景里的星星收集物；
- 玩家和星星之间的碰撞或重叠检测；
- 分数 UI；
- 一个简单的危险物或失败条件；
- 失败后的重新开始逻辑；
- 每个学习步骤都有清晰的小提交。

这个范围刻意保持很小。它足够形成完整游戏闭环，又不会让注意力被复杂美术、复杂架构或过早优化带偏。

## 教练模式

Codex 作为教练主导节奏，你作为有经验的开发者学习 Cocos。你不是被动看代码，而是要参与判断、操作、修改和复盘。

每一课按这个节奏进行：

1. 先确定一个很小、可运行的目标。
2. 点出这一课需要理解的 Cocos 概念。
3. 实现最小可工作的版本。
4. 解释代码在 Cocos 运行模型里的位置。
5. 让你做一个小观察、小修改或小预测。
6. 用清晰的提交记录保存这一课。

讲解以当前代码和编辑器操作为中心，避免脱离实战的大段抽象说明。除非你主动要求深入，否则不做死知识灌输。

## 项目结构

Cocos 项目位于 `learn-001`。

计划中的 `assets` 结构：

- `assets/scenes`：游戏场景。
- `assets/scripts`：TypeScript 组件和游戏逻辑。
- `assets/prefabs`：星星、危险物等可复用对象。
- `assets/textures`：需要时放简单视觉资源。

项目学习文档放在 `docs/superpowers/specs`。

## 课程节奏

第 1 课：让场景活起来。

- 创建或检查主场景。
- 添加玩家节点。
- 挂载 TypeScript 组件。
- 用键盘输入移动玩家。
- 学习重点：Node、Component、脚本挂载、生命周期、输入事件、delta time。

第 2 课：加入可收集物。

- 创建星星节点或 Prefab。
- 检测玩家是否碰到星星。
- 增加分数。
- 学习重点：Prefab、节点引用、简单碰撞或距离检测、UI Label。

第 3 课：生成和重置。

- 让星星出现在不同位置。
- 限制玩家不要移出边界。
- 收集后重置星星。
- 学习重点：坐标空间、随机位置、基础 GameManager 职责。

第 4 课：加入风险。

- 加入危险物或倒计时。
- 触发失败状态。
- 暂停或重置游戏逻辑。
- 学习重点：游戏状态、启停逻辑、场景重启。

第 5 课：打磨游戏闭环。

- 加入基础反馈、简单音效和开始/重开 UI。
- 学习重点：资源管理、UI 事件绑定、基础表现打磨。

## 代码组织

前期直接使用 Cocos 的组件模型，不引入大型框架，也不提前设计复杂架构。

初始组件规划：

- `PlayerController`：负责玩家输入和移动。
- `Star`：必要时负责星星自身行为。
- `GameManager`：负责分数、生成、失败状态和重开。
- `Hazard`：如果加入危险物，则负责危险物自身行为。

每个组件尽量只承担一个清晰职责。等某个脚本开始变得臃肿时，再在课程中自然重构，并解释为什么要拆。

## 调试方式

优先使用看得见、能教学的调试方式：

- 查看 Cocos Console 里的运行错误；
- 在编辑器里检查节点引用；
- 在 Inspector 里确认组件属性；
- 必要时加入临时日志说明流程；
- 课程结束后清理没有价值的噪音日志。

遇到常见问题时要顺手讲清楚，例如：节点引用没赋值、脚本没有挂到节点上、坐标理解错、生命周期时机不对。

## 验证方式

每一课结束时都要在 Cocos Creator 里手动运行验证：

- 场景能运行；
- 新增交互有效；
- Console 没有意外错误；
- 你能说清楚这一课改了什么、为什么这么改。

第一阶段不强求代码级自动化测试，因为核心价值在于熟悉 Cocos 编辑器、节点、组件和运行时绑定。等后面出现更复杂的纯 TypeScript 逻辑时，再考虑加入自动化测试。
